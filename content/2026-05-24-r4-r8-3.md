## R4 R8 테스트 3은 무엇을 확인하는가

### 성공 메시지와 실제 성공은 다르다

R4 R8 최종 안정화 테스트 3은 이름만 보면 내부 점검처럼 보입니다. 그런데 이 테스트가 확인하는 것은 꽤 실제적입니다. 여러 에이전트가 디스코드에서 응답하고, 사전점검을 남기고, 이미지 생성까지 했을 때 그 흐름이 실제 Draft 작성으로 안정적으로 이어지는지 보는 작업입니다.

이번 컨텍스트에는 Grok의 이미지 생성 완료 기록이 있습니다. `xai/grok-imagine-image` 모델로 이미지가 만들어졌고, 생성된 이미지 경로도 남아 있습니다. Claude7도 사전점검 완료 메시지를 남겼습니다. 겉으로 보면 작업이 꽤 많이 진행된 것처럼 보입니다. 그런데 여기서 바로 중요한 질문이 나옵니다. 이것이 본문 작성에 필요한 실제 입력으로 들어왔는가, 아니면 채널에 보이는 운영 로그로만 남아 있는가.

사실 자동화에서 가장 위험한 착시는 “보였다”를 “전달됐다”로 착각하는 것입니다. 디스코드 채널에 메시지가 보인다고 해서 작성 세션이 그 내용을 모두 받은 것은 아닙니다. 이미지가 생성됐다고 해서 그 이미지가 GitHub Pages에서 접근 가능한 위치에 복사된 것도 아닙니다. 검수 완료라는 말이 있다고 해서 실제 Draft를 읽은 검수라는 뜻도 아닙니다.

그래서 이 테스트는 모델의 글쓰기 실력을 보는 것이 아닙니다. 순서, 핸드오프, 상태 표시, 멈춤 조건을 보는 작업입니다. R4와 R8이 안정적이라는 말은 각 단계가 자기 역할을 알고, 다음 단계가 필요한 입력을 실제로 받으며, 발행하면 안 되는 상황에서 멈출 수 있다는 뜻이어야 합니다.

### 이번 입력에서 확인된 것과 확인되지 않은 것

이번 입력에서 확인된 것은 분명합니다. Grok은 이미지 생성 완료 로그를 남겼습니다. Claude7은 사전점검 완료 로그를 남겼습니다. 하네스 컨텍스트 번들은 이 기록을 본문 작성 세션으로 전달했습니다. 주제, 날짜, 요구사항, 발행 금지 조건도 함께 들어왔습니다.

그런데 확인되지 않은 것도 있습니다. Grok이 만든 이미지는 로컬 생성 경로에 남아 있을 뿐, 이 Markdown 안에서 바로 GitHub Pages용 이미지로 참조해도 되는지는 확인되지 않았습니다. 파일을 직접 쓰지 말라는 조건이 있으므로 이미지를 옮기거나 저장소에 복사할 수 없습니다. 따라서 본문에서는 이미지가 생성됐다는 운영 사실은 반영하되, 실제 배포 가능한 이미지 링크처럼 다루면 안 됩니다.

Claude7도 마찬가지입니다. Claude7 사전점검은 있었지만, 아직 이 Draft가 작성되기 전입니다. 그러니 이번 단계의 Claude7 기록은 본문 검수가 아니라 사전점검 기록입니다. 본문 품질을 확인하려면 이 Markdown이 Draft로 저장된 뒤, 그 저장된 Draft를 기준으로 다시 검수해야 합니다.

솔직히 이 구분이 귀찮아 보일 수 있습니다. 하지만 자동화에서는 이런 구분이 신뢰를 만듭니다. 무엇이 실제 산출물이고, 무엇이 사전 신호인지 구분하지 못하면 하네스는 금방 가짜 완료 상태가 됩니다.

## 사전점검은 시작 신호일 뿐이다

### Grok의 역할은 본문 작성이 아니다

이번 Grok 블록에는 역할 경계가 명확하게 적혀 있습니다. X 데이터, 이미지 생성, 트렌드 조사만 담당하고, 본문 작성과 구조 검수는 하지 않는다고 되어 있습니다. 이 문장은 아주 중요합니다. 에이전트가 많아질수록 역할 경계가 흐려지기 쉽기 때문입니다.

Grok이 이미지를 만들었다고 해서 본문까지 책임지는 것은 아닙니다. Grok이 사전점검 완료를 남겼다고 해서 블로그 초안이 완성된 것도 아닙니다. Grok은 자기 역할 안에서 필요한 신호를 남겼고, 다음 작업자에게 넘겼습니다. 여기까지가 Grok의 책임입니다.

사실 멀티 에이전트 하네스에서 좋은 에이전트는 모든 일을 다 하는 에이전트가 아닙니다. 자기 역할을 정확히 알고, 결과를 다음 단계가 쓸 수 있게 넘기는 에이전트입니다. 이미지 담당은 이미지를 만들고, 리서치 담당은 근거를 남기고, 작성 담당은 본문을 만들고, 검수 담당은 실제 Draft를 읽어야 합니다.

이번 테스트에서는 Grok의 이미지 생성 기록을 본문 맥락에 반영할 수 있습니다. 다만 실제 이미지 링크를 넣어 발행 가능한 척하면 안 됩니다. 파일을 직접 쓰지 않는 조건에서는 생성 파일을 정적 경로로 옮길 수 없기 때문입니다. 이런 멈춤이 바로 하네스의 안전장치입니다.

### Claude7의 사전점검은 본문 검수가 아니다

Claude7 블록에는 “Claude7 OK”와 사전점검 완료 기록이 있습니다. 이 기록은 하네스가 Claude7과 연결됐다는 신호로는 의미가 있습니다. 하지만 본문 검수로는 아직 의미가 없습니다. 이 Draft가 아직 없었기 때문입니다.

이번 하네스 사고에서 가장 크게 배운 것도 이 지점입니다. Claude7 검수 시점이 본문 전이라 무의미했습니다. 검수라는 단계 이름은 있었지만, 실제 본문이 없었습니다. 그러니 문장 흐름, 요구사항 누락, 길이 초과, 제목 불일치, 금지 조건 위반을 확인할 수 없었습니다.

그래서 흐름을 바꿨습니다. 먼저 Draft를 만듭니다. 그 Draft를 저장합니다. 그 다음 Claude7이 저장된 Draft를 기준으로 검수합니다. 검수 결과를 다시 본문에 반영합니다. 마지막으로 품질 게이트가 길이와 구조와 발행 제한을 확인합니다. 이 순서가 되어야 검수라는 말이 실제 의미를 가집니다.

그런데 사전점검이 쓸모없다는 뜻은 아닙니다. 사전점검은 요구사항이 들어왔는지, 역할 경계가 맞는지, 다음 단계가 무엇인지 확인하는 데 필요합니다. 다만 사전점검과 본문 검수를 같은 이름으로 묶으면 사고가 납니다. 이번 테스트 3에서는 이 구분을 더 분명히 해야 합니다.

## Draft 저장 후 검수로 바뀐 이유

### 검수 대상이 있어야 검수다

검수는 대상이 있어야 합니다. 너무 당연한 말인데 자동화에서는 자주 깨집니다. 검수 모델이 아무리 좋아도 읽을 Draft가 없으면 제대로 된 검수를 할 수 없습니다. 본문이 없으면 문장 흐름을 볼 수 없고, H2/H3 구조도 볼 수 없고, 글자 수 조건도 확인할 수 없습니다.

이번 하네스 사고는 이 단순한 원칙을 다시 확인시켰습니다. Claude7이 본문 작성 전에 검수를 하면, 실제로는 사전 의견이나 추정만 할 수 있습니다. 그런 결과를 최종 검수로 착각하면 품질 게이트가 무너집니다. 특히 블로그 자동화에서는 이 문제가 더 큽니다. 글은 길이, 구조, 제목, 태그, 이미지, 발행 조건까지 함께 봐야 하기 때문입니다.

Draft 저장 후 검수로 바꾸면 상황이 달라집니다. 검수자는 실제 파일이나 실제 Markdown 본문을 읽습니다. 어떤 문단이 중복되는지, 요구사항이 빠졌는지, 금지 표현이 있는지, 길이가 부족하거나 초과했는지 확인할 수 있습니다. 검수 결과도 더 구체적입니다.

사실 이 변화는 작은 순서 변경처럼 보입니다. 하지만 실제 운영에서는 큰 차이를 만듭니다. “검수 단계가 있다”에서 “검수할 실제 Draft가 있다”로 바뀌기 때문입니다. 자동화 품질은 이런 작은 차이에서 갈립니다.

### 사전점검, Draft, 검수의 이름을 분리해야 한다

하네스 안정화를 위해서는 단계 이름을 더 정확히 써야 합니다. “검수”라는 말을 너무 넓게 쓰면 안 됩니다. 사전점검, 본문 검수, 품질 게이트는 서로 다릅니다.

사전점검은 작성 전에 합니다. 요구사항이 들어왔는지, 역할 경계가 맞는지, 리서치나 이미지 같은 입력이 있는지 확인합니다. Draft는 작성 단계입니다. 저장 가능한 본문을 만듭니다. 본문 검수는 Draft가 생긴 뒤에 합니다. 실제 내용을 읽고 구조와 흐름과 누락을 확인합니다. 품질 게이트는 마지막입니다. 길이, H2/H3, frontmatter, 발행 금지 같은 조건을 확인합니다.

이렇게 이름을 나누면 운영 로그가 훨씬 선명해집니다. “Claude7 사전점검 완료”와 “Claude7 본문 검수 완료”는 다른 상태입니다. “Grok 이미지 생성 완료”와 “이미지 배포 가능”도 다른 상태입니다. “Draft 출력 완료”와 “파일 저장 완료”도 다릅니다.

솔직히 이런 구분은 처음에는 번거롭습니다. 하지만 문제가 생겼을 때 원인을 빨리 찾게 해줍니다. 어디서 멈췄는지, 무엇이 빠졌는지, 다음에 어떤 작업이 필요한지 보입니다. 안정화 테스트는 바로 이 선명함을 만드는 과정입니다.

## R4 R8 안정화에서 이미지 로그를 다루는 법

### 생성된 이미지와 게시 가능한 이미지는 다르다

이번 Grok 블록에는 생성된 이미지 경로가 있습니다. 로컬 경로로 보면 이미지가 실제로 생성된 것 같습니다. 그런데 이 경로는 블로그 독자가 접근할 수 있는 URL이 아닙니다. GitHub Pages에 올라간 정적 파일 경로도 아닙니다. 따라서 이 Markdown에 그 로컬 경로를 그대로 넣으면 게시 후 이미지가 깨질 가능성이 큽니다.

이 차이를 놓치면 시각 자료가 있는 것처럼 보이지만 실제 페이지에서는 보이지 않는 문제가 생깁니다. 하네스는 이런 상태를 구분해야 합니다. 이미지 생성 완료는 생성 완료입니다. 이미지 배치 완료는 별도입니다. 이미지 참조 검수도 별도입니다. 이 세 단계를 하나로 묶으면 안 됩니다.

사실 이미지가 들어간 블로그 하네스는 글만 있는 하네스보다 더 복잡합니다. 이미지 생성, 파일 이동, 경로 변환, 본문 삽입, 실제 파일 존재 확인, 페이지 렌더링 확인이 필요합니다. 파일을 직접 쓰지 말라는 이번 조건에서는 이 중 파일 이동과 삽입을 실제로 수행할 수 없습니다.

그래서 이번 Draft에서는 이미지 생성 사실을 운영 맥락으로만 다룹니다. 실제 Markdown 이미지 링크를 넣지는 않습니다. 이게 더 안전합니다. 보이는 것을 억지로 발행 가능한 자산처럼 꾸미지 않는 것이 하네스의 기본 태도입니다.

### visual pack은 품질 게이트의 별도 항목이어야 한다

이미지 묶음이 들어가는 작업에서는 visual pack 검수가 필요합니다. 이미지가 본문에 몇 개 들어갔는지, 참조 경로가 실제 존재하는지, 누락된 이미지가 있는지, 본문 분포가 자연스러운지 확인해야 합니다. 글의 품질과 이미지의 품질은 연결되어 있지만, 검수 항목은 분리되어야 합니다.

예를 들어 본문은 통과했는데 이미지 파일이 없으면 페이지 품질은 실패입니다. 반대로 이미지가 있어도 본문 길이가 기준을 벗어나면 발행 게이트는 실패할 수 있습니다. 그래서 하네스는 텍스트 게이트와 visual pack 게이트를 나눠 보는 편이 좋습니다.

이번 테스트 3에서는 이미지 생성 로그가 있지만, 실제 파일 배치와 본문 삽입은 하지 않습니다. 이유는 명확합니다. 사용자 요청이 파일을 직접 쓰지 말라는 것이고, 네이버나 인스타 자동 발행도 하지 말라는 것이기 때문입니다. 생성된 이미지를 다루는 작업은 별도 단계에서 해야 합니다.

그런데 이 로그는 쓸모가 있습니다. 다음 단계에서 파일을 배치할 때 어떤 이미지가 생성됐는지 알 수 있습니다. 또 이미지 생성 담당과 본문 작성 담당의 핸드오프가 작동하는지 확인할 수 있습니다. 안정화 테스트에서는 이런 흔적이 중요합니다.

## R4 R8의 입력과 출력 계약

### R4는 무엇을 받아야 하는가

R4가 안정적으로 작동하려면 입력 계약이 필요합니다. 이번 작업 기준으로 보면 R4는 주제, 날짜, 요구사항, 하네스 컨텍스트, Grok 블록, Claude7 블록, 발행 제한을 받아야 합니다. 이 중 하나라도 빠지면 판단이 흐려집니다.

주제는 “R4 R8 최종 안정화 테스트 3”입니다. 날짜는 2026-05-24입니다. 요구사항은 Markdown 본문, YAML frontmatter, 9500자에서 12500자 목표, H2마다 H3 포함, H3 개수는 H2 이상, 한자와 일본어 표현 금지, Claude7 사고 교훈 포함, 네이버와 인스타 자동 발행 금지입니다.

Grok 블록은 이미지 생성 완료와 사전점검 로그입니다. Claude7 블록은 사전점검 완료 로그입니다. 이 둘은 본문에 반영하되, 깊은 리서치나 본문 검수처럼 과장하면 안 됩니다. 특히 Claude7은 Draft 작성 전 단계이므로 본문 검수가 아닙니다.

이렇게 입력을 명확히 하면 R4는 안정적으로 다음 단계에 넘길 수 있습니다. 입력이 없으면 없는 대로 표시하고, 있는 입력은 역할에 맞게 반영합니다. 하네스가 해야 할 일은 빈칸을 지어내는 것이 아니라, 빈칸을 빈칸으로 인식하는 것입니다.

### R8은 무엇을 내야 하는가

R8이 최종 안정화 단계라면 출력 계약이 더 중요합니다. R8은 사용자가 요청한 형식으로만 결과를 내야 합니다. 이번 경우에는 저장 가능한 Markdown 본문만 출력해야 합니다. 코드펜스도 없어야 하고, 파일을 직접 쓰면 안 됩니다. 외부 발행도 하면 안 됩니다.

R8의 출력에는 frontmatter가 있어야 합니다. title은 주제와 맞아야 하고, date는 오늘 날짜여야 하며, tags가 있어야 합니다. 본문에는 H2와 H3 구조가 있어야 합니다. 각 H2마다 최소 하나의 H3가 있어야 하고, 전체 H3 개수는 H2 이상이어야 합니다.

또한 이번 하네스 사고 교훈이 반드시 들어가야 합니다. Claude7 검수 시점이 본문 전이라 무의미했고, Draft 저장 후 검수로 바꿨다는 내용입니다. 이 내용은 단순히 한 줄로 끼워 넣기보다, 왜 중요한지 설명되어야 합니다. 그래야 테스트 글이 운영 교훈으로 읽힙니다.

마지막으로 R8은 멈춤 조건을 지켜야 합니다. 네이버나 인스타 자동 발행을 하지 않습니다. 파일도 쓰지 않습니다. 이미지 생성 로그가 있어도 로컬 파일을 임의로 참조하지 않습니다. 이 멈춤까지 포함해야 안정화입니다.

## 품질 게이트는 마지막 방어선이다

### 길이와 구조는 기계적으로 확인해야 한다

블로그 하네스에서 품질 게이트는 마지막 방어선입니다. 사람이 보기에는 글이 자연스러워도, 요구사항을 놓치면 실패입니다. 이번 조건에서는 9000자 미만이거나 13000자를 넘으면 발행 게이트에서 실패합니다. 목표는 9500자에서 12500자입니다.

길이는 감으로 맞추기 어렵습니다. 모델은 종종 길게 씁니다. 또는 너무 짧게 끝냅니다. 그래서 품질 게이트가 필요합니다. 최종 Draft를 기준으로 글자 수를 보고, 필요한 경우 압축하거나 보강해야 합니다. 이때도 핵심 내용은 유지해야 합니다.

구조도 마찬가지입니다. H2마다 H3가 있어야 합니다. 전체 H3 개수는 H2 이상이어야 합니다. 이것은 단순한 형식 요구가 아니라 독자 흐름을 위한 장치이기도 합니다. H2만 길게 이어지면 글이 무겁고, H3가 있으면 세부 논점이 나뉩니다.

사실 구조 조건을 처음부터 염두에 두면 글이 더 안정됩니다. 먼저 큰 섹션을 만들고, 각 섹션마다 작은 질문을 둡니다. 그러면 글이 자동으로 흐름을 갖습니다. 하네스 테스트 글처럼 내부 개념이 많은 글에서는 이 구조가 특히 필요합니다.

### 금지 조건은 마지막이 아니라 처음부터 반영해야 한다

금지 조건은 마지막에 지우는 방식보다 처음부터 반영하는 편이 안전합니다. 이번 요청에는 한자와 일본어 표현 금지가 있습니다. 네이버와 인스타 자동 발행 금지도 있습니다. 파일 직접 쓰기 금지도 있습니다. 코드펜스 금지도 있습니다.

이런 조건을 마지막에 고치려 하면 놓치기 쉽습니다. 특히 외부 발행 금지는 행동 조건이므로, 한 번 실행하면 되돌리기 어렵습니다. 그래서 처음부터 “이번 작업은 초안 출력만 한다”는 경계를 세워야 합니다.

한자와 일본어 표현 금지도 주의가 필요합니다. 한국어 글에서 한자어 자체를 모두 피하라는 뜻은 아닐 수 있지만, 실제 한자 문자나 일본어 표현은 넣지 않는 것이 맞습니다. 제목, 본문, 태그, 이미지 설명에도 불필요한 외국어 표현을 줄이는 편이 안전합니다.

솔직히 이런 제한은 귀찮을 수 있습니다. 하지만 제한이 있으면 하네스의 행동 범위가 선명해집니다. 어디까지 하고 어디서 멈출지 알 수 있습니다. 자동화에서는 이 선명함이 품질입니다.

## 테스트 글을 실제 운영에 연결하는 법

### 운영 로그를 독자가 읽을 수 있는 말로 바꾸기

테스트 글은 자칫 내부 로그처럼 보이기 쉽습니다. JOBTAG, ACK, Grok, Claude7, R4, R8 같은 말이 많아지면 독자는 흐름을 놓칠 수 있습니다. 그래서 운영 로그를 그대로 노출하기보다, 의미를 풀어 써야 합니다.

예를 들어 JOBTAG는 작업을 구분하는 꼬리표입니다. ACK는 응답이 특정 작업과 연결됐다는 확인입니다. Grok은 이미지 생성이나 사전점검을 맡았습니다. Claude7은 사전점검을 남겼지만, 아직 본문 검수는 하지 않았습니다. 이렇게 설명하면 내부 용어도 읽을 수 있는 문장이 됩니다.

사실 기술 글에서 내부 용어를 완전히 숨길 필요는 없습니다. 다만 독자가 이해할 수 있게 맥락을 붙여야 합니다. “R4 R8 안정화”라는 표현만 던지면 좁은 운영자만 이해합니다. “사전점검과 최종 보정 단계가 같은 Draft를 기준으로 이어지는지 확인한다”라고 풀면 더 넓은 독자가 따라올 수 있습니다.

이번 글도 그래서 운영 로그와 블로그 문장 사이의 균형을 잡아야 합니다. 너무 내부용으로 쓰면 공개 글로 약하고, 너무 일반론으로 쓰면 테스트 의미가 사라집니다. 둘 사이에서 핵심은 순서와 핸드오프입니다.

### 초안과 발행본을 분리하기

테스트 글은 초안과 발행본을 분리해야 합니다. 지금 요청은 저장 가능한 Markdown 본문을 출력하는 것입니다. 파일을 직접 쓰지 않고, 네이버나 인스타 자동 발행을 하지 않는 조건입니다. 그러니 이 단계의 결과는 발행본이 아니라 Draft입니다.

Draft는 검수를 받아야 합니다. 특히 이번 사고 교훈에 따르면 Draft 저장 후 검수가 맞습니다. 지금은 본문을 출력하는 단계이고, 이 본문이 저장된 뒤에 Claude7이 실제 Draft를 읽어야 합니다. 그 검수 결과를 반영해 보정한 뒤에야 발행 가능성을 논할 수 있습니다.

그런데 종종 자동화는 이 단계를 건너뛰고 싶어 합니다. 초안이 잘 나왔으니 바로 발행하면 될 것처럼 느껴집니다. 하지만 테스트 글은 더 조심해야 합니다. 내부 로그가 남아 있거나, 이미지 경로가 배포 가능하지 않거나, 표현이 독자용으로 다듬어지지 않았을 수 있습니다.

그래서 초안과 발행본을 분리하는 것은 품질 문제이면서 안전 문제입니다. 초안은 실험과 검수의 대상이고, 발행본은 독자에게 공개되는 결과입니다. 이 둘을 섞지 않는 것이 하네스 운영의 기본입니다.

## 다음 테스트에서 더 줄일 수 있는 불안

### 상태 이름을 더 정확히 쓰기

다음 R4 R8 테스트에서 줄일 수 있는 불안은 상태 이름입니다. “OK”만으로는 부족합니다. 무엇이 OK인지 알 수 없기 때문입니다. 사전점검 OK인지, 이미지 생성 OK인지, Draft 작성 OK인지, 본문 검수 OK인지, 품질 게이트 OK인지 구분해야 합니다.

상태 이름을 정확히 쓰면 운영자가 덜 헷갈립니다. “Grok 이미지 생성 완료, 정적 경로 배치 전”처럼 쓰면 이미지가 만들어졌지만 아직 게시 가능한 상태는 아니라는 점이 보입니다. “Claude7 사전점검 완료, 본문 검수 전”이라고 쓰면 검수가 끝난 것이 아니라는 점이 보입니다.

이런 표현은 조금 길지만 유용합니다. 자동화가 커질수록 짧은 성공 메시지는 오히려 위험할 수 있습니다. 짧은 메시지가 나쁜 것은 아니지만, 상태를 충분히 담아야 합니다.

사실 하네스는 사람과 기계가 같이 읽는 운영 문서입니다. 사람은 의미를 보고, 기계는 상태를 봅니다. 둘 다 이해할 수 있는 이름을 쓰는 것이 안정화에 도움이 됩니다.

### 이미지 단계는 별도 게이트로 나누기

이번 테스트에는 이미지 생성이 포함됐습니다. 다음에는 이미지 단계를 더 분리하면 좋습니다. 이미지 생성, 이미지 저장소 배치, 본문 삽입, 경로 검수, 렌더링 확인을 각각 나누는 방식입니다.

이미지가 생성됐다는 말은 유용하지만, 최종 페이지에서 보인다는 뜻은 아닙니다. 특히 로컬 경로와 정적 사이트 경로는 다릅니다. 이 차이를 게이트로 나누면 깨진 이미지 문제를 줄일 수 있습니다.

예를 들어 이미지 생성 담당은 로컬 파일을 만들고, 배치 담당은 정적 폴더에 복사하고, 본문 담당은 배포 가능한 상대 경로를 넣고, 검수 담당은 파일 존재 여부와 렌더링을 확인하는 식입니다. 이렇게 나누면 어느 단계에서 문제가 생겼는지 바로 보입니다.

물론 이번 요청에서는 파일을 직접 쓰지 않으므로 이 단계를 실행하지 않습니다. 하지만 설계 원칙으로는 남겨둘 만합니다. R4 R8 안정화가 계속된다면 visual pack 게이트는 별도 항목으로 관리하는 편이 좋습니다.

## 결론: 테스트 3의 핵심은 더 분명한 멈춤이다

### 성공은 더 많이 하는 것이 아니라 정확히 멈추는 것이다

R4 R8 최종 안정화 테스트 3의 핵심은 더 많이 하는 것이 아닙니다. 정확히 멈추는 것입니다. Grok이 이미지를 만들었어도, 파일을 직접 쓰지 않는 조건에서는 정적 경로로 옮기지 않습니다. Claude7이 사전점검을 했어도, 본문 전 검수는 최종 검수로 보지 않습니다. 네이버와 인스타 자동 발행은 하지 않습니다. Draft만 출력합니다.

이 멈춤은 소극적인 행동이 아닙니다. 오히려 안정적인 자동화의 적극적인 조건입니다. 자동화는 빨리 움직일수록 멈춤 기준이 필요합니다. 멈춤 기준이 없으면 할 수 있는 일을 다 해버리고, 그중 일부는 사용자가 원하지 않은 일이 될 수 있습니다.

사실 좋은 하네스는 사용자의 요청을 넓게 해석해서 더 많이 하는 구조가 아닙니다. 요청된 범위 안에서 정확히 처리하고, 다음 단계가 필요하면 그 상태를 남기는 구조입니다. 이번 작업은 저장 가능한 Markdown 본문을 만드는 데서 끝납니다.

이렇게 멈추면 다음 단계가 선명해집니다. 이 Draft를 저장한 뒤 Claude7 본문 검수를 돌릴 수 있습니다. 검수 결과를 반영해 보정할 수 있습니다. 품질 게이트를 확인할 수 있습니다. 필요하다면 그 뒤에 발행을 논할 수 있습니다. 순서가 생깁니다.

### R4 R8 안정화는 반복 가능한 순서다

R4 R8 안정화는 한 번의 성공이 아니라 반복 가능한 순서입니다. 입력 확인, 사전점검 구분, Draft 작성, Draft 저장, 본문 검수, 보정, 품질 게이트, 발행 보류. 이 순서가 여러 번 반복되어도 흔들리지 않아야 합니다.

이번 테스트 3에서는 이미지 생성 로그가 추가됐습니다. 그래서 이미지 상태와 본문 상태를 구분하는 일이 더 중요해졌습니다. Grok의 작업은 의미가 있지만, 배포 가능한 이미지 삽입과는 별개입니다. Claude7의 사전점검도 의미가 있지만, 본문 검수와는 별개입니다. 이 구분이 안정화의 핵심입니다.

솔직히 자동화에서 가장 큰 차이는 모델 이름보다 운영 습관에서 납니다. Draft 없이 검수하지 않는 습관, 발행 요청 없이 발행하지 않는 습관, 로컬 파일을 배포 이미지처럼 쓰지 않는 습관, 성공 메시지보다 산출물을 보는 습관입니다. 이런 습관이 쌓이면 하네스는 점점 믿을 수 있어집니다.

이번 Draft는 그 원칙을 확인하는 테스트 기록입니다. 파일을 직접 쓰지 않았고, 네이버나 인스타에 자동 발행하지 않았고, 본문 전 Claude7 검수를 최종 검수로 착각하지 않았습니다. 다음 단계는 저장된 Draft를 기준으로 실제 검수를 하는 것입니다. 하네스는 마법이 아니라 순서입니다. 그리고 이번 테스트 3은 그 순서를 한 번 더 고정하는 작업입니다.
