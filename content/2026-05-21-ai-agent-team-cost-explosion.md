## 도입부: AI 에이전트 팀은 생각보다 빨리 비싸집니다

AI 에이전트 팀을 처음 만들 때는 보통 성능부터 봅니다. 누가 더 똑똑한지, 어떤 모델이 답변이 좋은지, 어떤 프롬프트가 더 멋지게 돌아가는지부터 보게 됩니다.

그런데 실제 운영에 들어가면 가장 먼저 부딪히는 건 성능이 아니라 비용입니다. 특히 OpenClaw + Discord 같은 멀티 에이전트 구조를 쓰면, 에이전트가 잘 돌아가는 순간 오히려 돈도 같이 잘 나갑니다. 이유는 단순합니다. 대화가 많아지고, 컨텍스트가 길어지고, 같은 내용을 여러 번 재전송하고, 검토와 재검토가 반복되기 때문입니다.

X에서 Grok가 묶어 정리한 사례들을 보면 숫자가 꽤 극단적입니다.

- 하루에 300달러가 나간 사례
- 3일 만에 1,000달러를 쓴 사례
- 반대로 구조를 잘 잡아서 월 10~70달러 수준으로 운영한 사례

이 차이는 모델 자체의 성능 차이보다, 어떤 일을 어떤 모델에게 맡기고, 어디서 반복을 끊느냐의 차이입니다.

## 왜 비용이 이렇게 쉽게 폭발할까

가장 흔한 원인은 다 좋은 모델로 하면 되겠지라는 생각입니다. 실제로는 그 반대입니다. 모든 작업을 고가 모델에게 맡기면 가장 빨리 망가집니다.

### 1. 모든 걸 고가 모델이 처리함

처음에는 편합니다. 하나의 모델이 기획도 하고, 수정도 하고, 검토도 하니까요.

문제는 쉬운 작업과 어려운 작업이 섞여 있다는 점입니다.

- 파일명 정리
- 문장 다듬기
- 링크 검사
- 한자 검출
- 길이 확인
- 실제 설계 판단

이걸 전부 같은 모델이 생각해서 처리하면, 생각이 필요한 부분보다 반복이 더 많아집니다. 비용이 빠르게 올라가는 건 당연합니다.

### 2. 컨텍스트를 매번 다시 보냄

에이전트 팀은 종종 같은 맥락을 계속 다시 보냅니다. 이전 채팅, 파일, 로그, 작업 히스토리까지 다시 넣다 보면 실제로는 별것 아닌 검토에도 토큰이 계속 쌓입니다.

컨텍스트가 길어질수록 모델은 똑똑해지기보다 비싸집니다. 그래서 요약이 중요하고, 작업 단위를 쪼개는 게 중요합니다.

### 3. 에이전트가 서로 대화만 하다가 끝남

멀티 에이전트 구조에서 의외로 많이 생기는 비용이 유령 비용입니다. 에이전트끼리 heartbeat를 주고받고, 확인하고, 다시 확인하고, 같은 정보를 재검토하는 흐름입니다.

겉으로는 일이 굴러가는 것 같지만 실제로는 돈만 줄어드는 경우가 많습니다.

### 4. 반복 가능한 일을 LLM으로만 해결하려고 함

토큰을 써서 해결할 수 있는 일이 있고, 코드로 끝내야 하는 일이 있습니다. 예를 들어:

- 한자 검출
- 분량 측정
- 금지어 카운트
- 특정 문자열 패턴 검색

이런 건 LLM이 아니라 grep, wc, sed 같은 도구가 더 정확하고 훨씬 저렴합니다.

## 실제로 비용을 줄인 사람들은 뭘 했나

Grok가 모아둔 후기들에서 공통적으로 보이는 패턴은 꽤 분명합니다.

### 1. 모델 라우팅을 나눴다

가장 효과가 큰 건 이겁니다.

- 오케스트레이션은 저렴한 모델
- 복잡한 추론만 비싼 모델
- 단순 검수는 도구

이렇게 층을 나누면 비용이 확 줄어듭니다. 많은 사람들이 80~90% 절감 사례를 이야기하는 이유도 결국 여기에 있습니다.

### 2. Skills를 적극적으로 만들었다

반복 작업을 LLM 호출로 처리하지 않고, 스킬이나 스크립트로 바꾸면 비용이 급격히 줄어듭니다.

핵심은 에이전트가 매번 생각하지 않아도 되는 일을 찾아내는 겁니다. 이걸 코드로 바꾸면 토큰이 줄고, 결과도 일정해집니다.

### 3. Prompt Caching을 썼다

시스템 프롬프트나 긴 고정 컨텍스트가 반복될수록 캐싱의 효과가 커집니다. 특히 동일한 구조의 작업을 계속 돌릴 때는 차이가 큽니다.

같은 설명을 매번 새로 보내지 않도록 만드는 것만으로도 체감 비용이 꽤 달라집니다.

### 4. 컨텍스트 요약을 생활화했다

긴 대화 전체를 유지하기보다, 핵심만 요약해서 넘기는 쪽이 낫습니다.

요약이 없으면 모델은 과거 로그를 다시 읽느라 돈을 쓰고, 요약이 있으면 현재 과제만 보게 됩니다.

### 5. 예산 상한을 걸었다

이건 실무에서 정말 중요합니다.

- 작업별 상한
- 일일 상한
- 모델별 사용 제한

이 셋이 없으면 멀티 에이전트 팀은 생각보다 쉽게 예산을 초과합니다. 좋은 결과가 나올 때까지를 방치하면 비용은 끝없이 늘어납니다.

## 비용이 터지는 구조와 줄어드는 구조

정리하면 아주 단순합니다.

### 비용이 터지는 구조

- 모든 작업을 고가 모델에 맡김
- 긴 컨텍스트를 계속 재전송함
- 에이전트끼리 쓸데없는 확인을 반복함
- 반복 가능한 작업도 전부 추론으로 해결하려고 함

### 비용이 줄어드는 구조

- 저가 모델과 고가 모델을 분리함
- 도구로 처리할 수 있는 일은 코드로 처리함
- 긴 대화는 요약해서 넘김
- 작업마다 종료 조건과 예산 상한을 둠

결국 핵심은 하나입니다. 에이전트의 숫자보다, 에이전트가 어떤 일을 얼마나 반복하느냐가 비용을 결정한다는 점입니다.

## 지금부터 바로 적용할 수 있는 체크리스트

멀티 에이전트 팀을 운영한다면 아래 다섯 가지부터 확인해보면 좋습니다.

- 이 작업은 정말 LLM이 생각해야 하나, 아니면 스크립트로 끝나나
- 이 모델은 지금 너무 비싼 자리에 쓰이고 있지 않나
- 같은 컨텍스트를 계속 다시 보내고 있지 않나
- 검토 루프가 과하게 길어져 있지 않나
- 작업별 또는 일별 비용 상한이 있나

이 다섯 개만 정리해도 꽤 많은 비용을 줄일 수 있습니다.

## 결론: 강력한 팀보다 지속 가능한 팀이 중요합니다

AI 에이전트 팀은 멋있게 보이는 순간보다 오래 버틸 수 있는 순간이 더 중요합니다.

처음에는 성능이 중요해 보이지만, 운영에 들어가면 결국 비용이 구조를 드러냅니다. 비용이 빨리 나간다면 그건 단순히 모델이 비싸서가 아니라, 역할 분리와 반복 제어가 덜 되어 있다는 뜻입니다.

그래서 저는 이 주제를 볼 때 늘 같은 결론으로 돌아옵니다.

좋은 AI 팀은 더 똑똑한 팀이 아니라, 덜 헷갈리고 덜 반복하는 팀입니다.

OpenClaw + Discord 같은 구조를 쓸 때도 마찬가지입니다. 역할을 나누고, 도구로 끝낼 일은 도구로 끝내고, 비싼 모델은 정말 필요한 순간에만 쓰는 것. 그게 결국 비용을 지키는 가장 현실적인 방법입니다.
